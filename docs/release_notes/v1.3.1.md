# v1.3.1-release-state-convergence-and-repo-hygiene

## 1. Summary

v1.3.1 is a control-plane and repository-hygiene patch. It makes the canonical JSON tracker, release documents, validation contracts, GitHub evidence, and external roadmap handoff describe one machine-readable state. It removes the accidental tracked root file `tatus --short` and adds a tracked-entry hygiene gate so command output, generated evidence, dependencies, and private runtime/user-data paths cannot silently enter a release candidate.

No product feature, route, response shape, Reader behavior, write workflow, storage format, migration, or runtime dependency is added.

## 2. Rationale

PR #2 and its hosted CI were complete, but current documents and tests still enforced the earlier provisional state. A separate post-merge `main` run also existed but had not been recorded. At the same time, a console-output file had been committed at repository root, and the external tracker required manual interpretation rather than a deterministic handoff. This patch separates evidence scopes and turns those assumptions into validated state transitions.

## 3. Files and contracts changed

- Runtime/version surfaces: `config/contact.py`, `frontend/package.json`, `frontend/package-lock.json`, frontend version labels, and README.
- Canonical state: `docs/tracker_sync_status.json` schema 2.0 with independent PR CI, post-merge CI, Streamlit regression, clean-PC restore, tag, and publication states.
- Release documents: README, roadmap, backlog, release checklist, regression checklist, and v1.3.0/v1.3.1 release notes.
- Validation contracts: conditional release-state invariants, contradiction tests, repository-hygiene tests, tracker-export tests, and smoke integration.
- New utilities: `scripts/check_repo_hygiene.py` and `scripts/export_tracker_status.py`.
- Deleted artifact: top-level `tatus --short`.

## 4. Repository hygiene behavior

The hygiene check uses `git ls-files` and inspects tracked entry names only. It does not enumerate or read ignored data. Narrow rules reject:

- the exact accidental filename `tatus --short`;
- root `.log`, `.out`, `.err`, and `.trace` artifacts;
- filenames containing shell flags, shell metacharacters, or common command fragments;
- generated validation-summary/result names that belong under ignored `artifacts/`;
- tracked dependency, test-temporary, secret, and private runtime/user-data paths.

The tracked `.gitkeep` placeholders under `data/`, `papers/`, `notes/`, and `exports/` are explicitly allowed. The check is part of `scripts/smoke_check.py`, so the normal local release gate and GitHub Actions Python job execute it.

## 5. External tracker handoff

The canonical tracker contains a versioned mapping for Drive roadmap tasks `R-001` through `R-025`. Every row uses one of `COMPLETED`, `SUPERSEDED`, `DEFERRED`, `PARTIAL`, or `NOT VERIFIED` and includes short repository-grounded evidence, a disposition/replacement milestone, and an ISO verification date.

`scripts/export_tracker_status.py` uses only the Python standard library. It sorts rows by task ID and writes deterministic UTF-8 CSV with exactly these columns:

`task_id,status,evidence,disposition,last_verified`

The default output is ignored `artifacts/tracker_status.csv`; `--output` selects another path. The export contains no Drive API/OAuth integration, network access, command output, user path, environment value, or private library metadata.

## 8. Compatibility and data safety

The local-first, single-user architecture is unchanged. All five GET-only API routes and response shapes, managed-root PDF containment, byte-range delivery, same-origin bridge behavior, `paper_id` generation, Streamlit write workflows, CSV/Markdown/JSON formats, projects, tags, snapshots, restore behavior, and caches remain unchanged. Tests use repository sources and disposable fixtures; they do not read real user library data.

## 9. Known limitations

- A genuinely clean-PC setup and restore rehearsal has not been performed.
- No v1.3.x tag or GitHub release is approved or published.
- Browser-native PDF presentation still varies by browser and does not provide application-owned annotation/highlight controls.
- Reader request count and index-read latency remain unmeasured; caching and PDF.js remain separately scoped future work.
- `npm ci` reports 12 dependency audit findings (2 low, 2 moderate, 8 high); dependency upgrades remain outside this bounded control-plane patch.

## 10. Publication state

This patch does not create a commit, push a branch, open or merge a pull request, create a tag, or publish a GitHub release. Those actions require separate explicit instruction.
