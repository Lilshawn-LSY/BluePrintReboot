# v1.3.0-reader-pdf-readonly-vertical-slice

## 1. Summary

v1.3.0 adds the first safe end-to-end read-only Reader/PDF vertical slice. Paper Detail now links by stable `paper_id` to a dedicated Reader route, and the browser receives a managed local PDF through a contained GET-only FastAPI endpoint and streaming same-origin bridge.

## 2. PDF endpoint and path-security contract

`GET /papers/{paper_id}/pdf` resolves the indexed paper record rather than accepting a path from the request. The resolved target must remain within the canonical managed papers directory, have a `.pdf` suffix, and be a regular file. Symlink or traversal resolution outside that root, non-PDF targets, and otherwise invalid paths are rejected.

The endpoint returns `application/pdf` inline through Starlette's file response, preserving streaming and byte-range behavior without base64 encoding or whole-file application buffering. Its controlled outcomes are 404 for an unknown paper, 404 for a missing managed PDF, 409 for an unsafe or invalid managed path, and 503 for a temporarily unavailable file. Error bodies and headers do not expose absolute filesystem paths.

## 3. Same-origin bridge

The frontend bridge adds only the exact `papers/{paper_id}/pdf` GET shape to its existing allowlist. It forwards a Range request for that shape, streams the upstream binary body, and copies only safe PDF response headers: content type, content length, content range, accept ranges, safe cache validators, last modified, and content disposition. Arbitrary paths and all write methods remain rejected, and the browser never receives the FastAPI origin.

## 4. Reader route and UI

`/papers/{paper_id}/reader` loads existing paper-detail metadata, presents a compact citation and lifecycle context, links back to Paper Detail, and displays the same-origin PDF URL with the browser-native PDF viewer. Paper Detail exposes Open Reader only from stable identity and presents an unavailable action when the paper has no managed PDF reference.

The Reader explicitly represents metadata loading, PDF loading, unknown paper, missing PDF, API unavailable, PDF response unavailable, archived paper, successful display, and browser-native viewer fallback. It remains navigable while the API is offline and contains no note editor or write action. Notes and every mutation workflow remain in Streamlit.

## 7. Compatibility and data safety

The local-first, single-user architecture and IPv4-loopback service bindings are unchanged. Existing API response shapes, `paper_id` generation, storage and note formats, projects, tags, snapshots, restore, extracted-text caches, Streamlit Reader/write behavior, and user data are unchanged. Automated PDF checks use disposable temporary indexes and files only.

## 8. Known limitations

- PDF presentation depends on browser-native PDF support and browser policy.
- Native rendering does not provide application-owned annotations, highlights, bookmarks, text-selection persistence, or consistent cross-browser controls.
- GitHub-hosted CI and clean-PC restore remain open release gates.

## 9. Deferred PDF.js work

PDF.js is intentionally not included. Renderer selection, dependency and security review, worker/assets packaging, accessibility, performance, and fallback behavior require a separately approved scope after this native vertical slice is validated.

## 10. Recommended next milestone

Complete v1.3.0 hosted-CI and clean-PC restore evidence, then harden native Reader behavior and evaluate PDF.js only if its benefits justify the added dependency and maintenance surface.
