# BluePrintReboot v1.0.0-foundation Release Notes - Draft

> Draft only. This is not a published GitHub Release and does not represent a final release commitment.

## Direction

BluePrintReboot is a local-first research paper library for readers who want PDFs, metadata, notes, and research organization to remain understandable and portable on their own machine. The v1.0.0-foundation direction keeps the Streamlit interface and stable `paper_id` identity model while bringing the existing workflows into a coherent daily-use experience. It is a stable foundation target, not a claim that the product is finished.

## Current Highlights

- **Local paper library** - `papers/` remains the canonical managed PDF location, with metadata in `data/paper_index.csv`.
- **Metadata and DOI workflow** - local DOI extraction, manual editing, and optional Crossref preview/acceptance with polite access and classified diagnostics.
- **Full-text extraction cache** - user-triggered extraction, backend diagnostics, SHA-256 freshness checks, and safe stale-cache recovery.
- **Reader Workspace** - PDF reading, Markdown notes, reading status, priority, and tags in one workspace.
- **Structured note blocks** - summaries, claims, methods, evidence, questions, ideas, and limitations stored by stable paper identity.
- **Project linking** - papers and note blocks linked into lightweight research projects with relationship types and notes.
- **Tag governance** - deterministic suggestions, canonical tags, aliases, audits, and previewed merges.
- **Paper Hygiene** - previewed human-readable filenames without changing `paper_id` or linked data.
- **Drive Inbox Import** - explicit copy-only import from a Drive-synced local folder into the canonical library.
- **Backup Snapshot** - light and full timestamped archives with checksummed manifests.
- **Library Health Check** - read-only diagnostics for missing, duplicate, orphaned, incomplete, stale, and noncanonical records.

## v1.0.10 through v1.0.15 Cumulative Summary

The post-v1.0.9 stabilization cycle focused on developer repeatability, scan predictability, deterministic tagging, profile-derived evidence, and PDF profile extraction repair:

- **v1.0.10 Developer bootstrap** - added Windows setup/check/run scripts and a launcher without changing app behavior or user data lifecycle.
- **v1.0.11 Scan/enrich/import guard** - made scan cheap and local by default, moved DOI/Crossref work to explicit metadata assist actions, preserved accepted metadata during scans, and blocked duplicate external note imports unless force re-import is explicitly selected.
- **v1.0.12 Tag Book v2** - moved canonical tag governance into `config/tag_book/` with aliases, method lexicon candidates, normalization, blocked terms, candidate patterns, validation, evidence-bearing grouped suggestions, and legacy config compatibility.
- **v1.0.13 PaperTextProfile minimal** - added rebuildable paper profile caches from index metadata, cached extracted-text abstract fallback, Reading Note sections, and structured note blocks so tag suggestions can show richer source evidence without changing source-of-truth storage.
- **v1.0.14 Tag quality hygiene** - cleaned generated candidate phrases, scored them by source quality, rejected low-quality or duplicate candidates, separated known canonical suggestions from generated candidates, and kept rejected candidates non-selectable.
- **v1.0.15 PDF profile extraction repair** - cleaned cached/full PDF text front matter to extract title, authors, DOI, abstract, keywords, article type, section headings, and warnings; used profile data to fill blank metadata preview fields; and added explicit `pdf_abstract`, `pdf_keywords`, and `pdf_section_headings` tag sources.

No FastAPI layer, frontend migration, packaging change, runtime data migration, paper identity redesign, OCR, external ontology integration, LLM/API tagging, automatic candidate promotion, or new destructive repair behavior is included in this cycle.

## Safety and Portability

- Runtime library data remains outside Git.
- Existing paper IDs are never regenerated by maintenance workflows.
- Destructive actions require confirmation where implemented.
- Inbox import copies rather than moves source files and never overwrites a managed PDF.
- Snapshot creation is additive; restore remains a documented manual process.
- The readiness smoke script does not delete, move, rename, restore, overwrite, or create personal library data.
- The v1.0.16 roadmap/release evidence sync is documentation and validation only; the app runtime version remains at the v1.0.15 baseline unless a separate version bump is approved.

## Known Gaps

The v1.0 draft does not include OCR, visual highlighting, coordinate annotations, Google Drive API/OAuth, Zotero integration, relation graphs, FastAPI/frontend migration, or an automated destructive restore workflow.

Remaining gaps after v1.0.16:

- Reader PDF rendering and Streamlit rerun boundaries still need stabilization.
- Duplicate PDF merge/remove semantics remain deferred.
- Orphan note and note-block reattach/export/delete workflows remain deferred.
- Extracted full-text `.txt` cache writes are not yet atomic.
- PDF profile extraction depends on readable cached/full extracted text and does not perform OCR, image parsing, figure/table extraction, or full methods/results extraction.
- Generated tag candidates remain preview/select-only and are not automatically promoted to the Tag Book.
