# v1.4.0-pdfjs-reader-foundation

## Summary

v1.4.0 replaces the web Reader's native browser viewer as the primary path with a controlled PDF.js canvas foundation. It keeps the existing read-only, stable-`paper_id`, managed-root PDF endpoint and same-origin bridge unchanged. Page navigation, bounded zoom, explicit loading/error/retry states, cleanup, safe diagnostics, and a conditional native fallback are included. Annotation, text/search layers, writes, storage migration, and broader UI work are not included.

## Reader architecture

- `PdfJsReader` is a client component and owns the canvas, semantic toolbar, loading/error states, retry, and fallback selection.
- `PdfReaderController` owns document-load cycles, page and zoom bounds, render cancellation, safe error classification, cleanup, and bounded diagnostics. It is framework-independent and tested with module-boundary PDF.js mocks.
- `pdfjs-adapter.ts` checks the browser boundary before dynamically importing `pdfjs-dist`. Server rendering does not import browser-only PDF.js code.
- The authoritative load uses the existing `/api/blueprint/papers/{paper_id}/pdf` same-origin route. The former separate one-byte Range probe is removed.

## Package and worker strategy

`pdfjs-dist` is pinned at `6.1.200` in `package.json` and `package-lock.json`. The client adapter dynamically imports `pdfjs-dist/build/pdf.worker.min.mjs?url` and assigns the emitted URL to `GlobalWorkerOptions.workerSrc`. Vinext/Vite therefore emits a hashed repository-built worker asset during the production build. There is no CDN, copied minified vendor bundle, arbitrary worker URL, or second PDF rendering library; once dependencies are installed, the build and runtime are local-first.

## Range delivery and security compatibility

No backend route or response shape changed. PDF.js reuses the existing GET-only endpoint and bridge. Stable `paper_id` resolution, managed-root containment, missing-paper/file behavior, full HTTP 200 delivery, HTTP 206 byte ranges, `Accept-Ranges`, `Content-Range`, exact lengths, malformed Range 400, and unsatisfiable Range 416 remain covered by disposable tests. The frontend accepts no path or arbitrary PDF URL and exposes no local filesystem path.

## Reader behavior

- The first page renders into one canvas after the PDF.js loading task resolves.
- Previous, next, and direct page entry clamp to the document bounds; first/last disabled states are visible.
- Zoom out, zoom in, and reset use a 50%-300% range in 25% steps and rerender only the active page.
- Loading and page-render progress are visible; safe unavailable, missing, load, and render failures offer retry.
- The native `<object>` viewer is mounted only after explicit fallback selection following a PDF.js failure. It is labeled, uses the same managed URL, and can return to PDF.js through retry.
- Buttons and input are keyboard-operable and labeled. The canvas has a page-specific accessible fallback description.

## Lifecycle and performance observability

One loading task is created per authoritative load cycle. Page navigation and zoom reuse the loaded document. New renders cancel or supersede active stale renders. Retry, fallback, `paper_id`/URL change, route exit, and component unmount clean the active render/page, loaded document, loading task, and canvas state. Cancellation promises are handled locally; errors are not globally suppressed.

When both a non-production build and `NEXT_PUBLIC_BLUEPRINT_READER_DIAGNOSTICS=1` are present, a small diagnostics disclosure shows document-load count/duration, page-render count, render-cancellation count, first-page render duration, and safely inferred request/Range mode from Resource Timing. It is disabled by default, uploads nothing, persists nothing, and contains no paper title, DOI, author, note content, URL, path, or PDF content. Real-browser request counts and Range use still require manual inspection because Resource Timing status visibility varies by browser.

## Known limitations and explicit non-goals

The Reader renders one page at a time and does not add text selection/search, text or annotation layers, highlights, comment pins, note-to-page links, thumbnails, continuous scrolling, persistent position, OCR, LLM features, mobile redesign, or write behavior. Full screen-reader PDF accessibility is outside this foundation; the canvas supplies only a page description. Resource Timing may not expose response status in every browser, so diagnostics can report `status-unavailable`.

Storage formats, paper identity, managed-root security, API routes and response shapes, CSV/note/project/tag/snapshot/cache formats, local-first single-user architecture, and every Streamlit write workflow remain unchanged. No user PDF, real note, or private library index is used by the automated tests.
