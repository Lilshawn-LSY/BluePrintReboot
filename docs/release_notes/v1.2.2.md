# v1.2.2-runtime-and-release-evidence-closure

## 1. Summary

v1.2.2 closes inconsistencies in the local frontend launch contract, workflow execution support, release evidence, version surfaces, checklists, and tracker handoff. It adds no product feature.

## 2. Scope and non-goals

The scope is release engineering and evidence closure only. Reader/PDF frontend work, new API routes, write APIs, UI redesign, deployment, packaging, migrations, authentication, cloud sync, telemetry, dependency upgrades, and restore automation remain out of scope.

## 3. Root cause of the localhost/127.0.0.1 discrepancy

Vinext 0.0.50 accepts `--hostname`/`-H`, not `--host`. The previous launcher forwarded `npm run dev -- --host 127.0.0.1 --port 3000`; Vinext ignored the unknown host pair, honored the port, and fell back to `localhost`. On the managed Windows environment, localhost resolves to IPv6 loopback `::1` before `127.0.0.1`, so Vinext listened on `::1:3000` and the IPv4 URL was not reachable.

The diagnosis was reproduced on 2026-07-18: the unsupported form listened on `::1` and only localhost returned HTTP 200; the supported `--hostname 127.0.0.1` form listened on `127.0.0.1` and both hostname probes returned HTTP 200. Each temporary process was stopped after the probe.

## 4. Canonical frontend URL and bind behavior

The canonical browser URL is `http://127.0.0.1:3000`. The launcher binds explicitly to IPv4 loopback with Vinext's supported `--hostname 127.0.0.1` option. It does not bind to `0.0.0.0`, bare `::`, a LAN address, or any external interface.

## 5. Launch diagnostic changes

Before handing control to the foreground server, `scripts/run_frontend.ps1` prints the configured bind address, configured port, canonical browser URL, Node version, npm version, Node source, exact post-launch HTTP probe, and an address-family inspection hint. Missing runtime, unsupported Node, missing package metadata, missing dependencies, and nonzero server exit retain distinct messages. Portable `-NodeHome` handling and temporary PATH restoration remain in the shared resolver.

## 6. Workflow trigger changes

The primary test workflow retains `push` and `pull_request`, separate Python/frontend jobs, Python smoke, full pytest, `npm ci`, lint, and frontend build/tests. It adds an empty `workflow_dispatch` trigger for deliberate hosted execution. The committed and pushed workflow definition is not itself hosted-CI evidence.

The local-first, single-user, localhost-only architecture is preserved. The four GET-only API route shapes, same-origin bridge, Streamlit workflows, storage formats, `paper_id`, notes, note blocks, project links, tags, snapshots, restore, extracted text, and ignore rules are unchanged. Automated checks use repository tests and disposable fixtures; no personal library content is included in evidence.

## 11. Known limitations

- Hosted CI remains open unless an actual relevant GitHub Actions run is available.
- Clean-PC restore remains a user-performed gate.
- `npm ci` reported 11 audit findings (2 low, 3 moderate, 6 high); dependency changes remain outside this bounded release.
- The frontend has no Reader/PDF vertical slice; Streamlit remains the PDF and write interface.
