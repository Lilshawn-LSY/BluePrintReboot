# BluePrintReboot v1.0.20 Safety Release Foundation

## What Changed

- Added typed corrupt/invalid JSON store errors and shared JSON read helpers for app-owned JSON persistence.
- Kept JSON saves consolidated on same-directory temporary files with flush, fsync, and atomic replacement.
- Updated Library Health Check to surface corrupt JSON, backup snapshot concerns, severity/category, meaning, and recommended next action.
- Made Backup Snapshot manifests explicit about included private library data, excluded source/runtime/cache files, and intentionally excluded regenerable extracted-text/profile caches.
- Added clearer Streamlit status, success, warning, error, and details-expander feedback around backup and health operations.
- Bumped the app version marker to `1.0.20`.

## Why It Changed

This release reduces the risk of silent local data loss during maintenance. Corrupt JSON should be preserved and repaired or restored deliberately, not overwritten by a fallback save path. Backup snapshots should clearly separate private library data from source code, virtual environments, secrets, logs, and regenerable caches.

## Validation Commands

```powershell
python scripts\smoke_check.py
python -m pytest -q
```

## Manual Smoke Checklist

- Open Settings and run Library Health Check.
- In a disposable workspace, confirm corrupt JSON is reported with path, issue summary, and recovery-safe next action.
- Create a light Backup Snapshot and inspect `manifest.json` for app version, included files, excluded policy, counts, and checksums.
- Create a full Backup Snapshot only after confirming PDF inclusion, then verify managed PDFs are included.
- Open a Reader paper, edit/save the Reading Note, and confirm the Reader note UX is unchanged.
- Confirm no user PDFs, notes, `paper_index.csv`, config secrets, `.venv`, or local caches are staged for Git.

## Known Limitations

- Backup restore remains manual and should be performed while the app is stopped.
- Extracted-text and paper-profile caches are excluded from snapshots by default because they are regenerable.
- The app reports corrupt JSON but does not auto-repair or quarantine it.
- Reader note UX refactor, hash-performance optimization, PDF rerender optimization, cloud sync, and frontend/API migration remain deferred.
