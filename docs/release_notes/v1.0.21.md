# BluePrintReboot v1.0.21 Reader Performance Polish

## What Changed

- Added conservative PDF hash metadata reuse using stored SHA-256, file size, and modified time.
- Updated scan, index backfill, Health Check duplicate detection, lifecycle repair, missing-PDF reconnect, and extracted-text freshness checks to avoid repeated full-file hashing when the file signature is unchanged.
- Preserved safety rules for duplicate and reconnect workflows: no automatic paper merge, no hash-only silent merge, and explicit confirmation remains required for mismatch repairs.
- Made Reader note draft state explicit with a saved/loaded baseline, safe reload behavior, pending header refresh state, and an explicit header-refresh action for unsaved drafts.
- Prevented metadata header refresh and note reload actions from silently overwriting unsaved Reading Note text.
- Added concise Streamlit feedback for note save, note reload, skipped reload, pending/applied header refresh, local PDF scans, inbox scans, and PDF import.
- Bumped the app version marker to `1.0.21`.

## Why It Changed

Large local PDF libraries should not rehash every unchanged file during routine maintenance. Reader metadata updates should also feel predictable: edits in progress stay in the draft until the user saves, reloads a clean draft, or explicitly applies a pending header refresh.

## Validation Commands

```powershell
python scripts\smoke_check.py
python -m pytest -q
```

## Manual Smoke Checklist

- Add or modify a disposable PDF under `papers/`, run **Scan papers (local sync)**, and confirm the index updates without duplicate rows.
- Run **Library Health Check** and confirm duplicate/missing PDF sections remain review-only and confirmation-gated.
- Open a Reader note, type unsaved text, save it, and confirm the saved notice appears.
- With unsaved Reader text present, use metadata editing to change the title/tags and confirm the note body is preserved and a header refresh is offered.
- Confirm **Reload** keeps an unsaved draft and reports that reload was skipped.
- Confirm no PDFs, notes, `paper_index.csv`, config secrets, `.venv`, exports, or local caches are staged for Git.

## Known Limitations

- `paper_id` remains path-derived; SHA-256 supports repair and duplicate review but is not the primary identity.
- Hash metadata reuse depends on file size and modified time staying accurate on the local filesystem.
- Unindexed PDFs still require full hashing when duplicate detection needs content identity and no trusted metadata exists.
- Reader notes still use the existing Streamlit text area and Markdown file format; there is no new editor component or PDF viewer architecture.
- Automatic paper merge, destructive cleanup, cloud sync, FastAPI migration, React/Next migration, PDF.js integration, and LLM/API features remain deferred.
