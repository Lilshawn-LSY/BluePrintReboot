# BluePrintReboot v1.0.24 Reader Validation and Parity Closure

## Behavior Changes

- Combined reading status and priority into one explicit **Apply reading settings** form action.
- Avoided metadata writes and explicit reloads when submitted settings are unchanged.
- Removed safe redundant application-triggered reruns for toolbar draft insertion, profile rebuild, tag preview, and opening structured-block edit.
- Preserved all paper-scoped draft, baseline, pending-event, notice, active-paper, page, and renderer state.
- Added the Reader action/rerun contract and future frontend parity checklist.

## Codex-Performed Disposable Checks

- Automated tests used only disposable workspaces.
- Settings payload construction, unchanged-submit behavior, note-state preservation, and paper-scoped renderer keys were checked automatically.
- No real Reader library, note, PDF, index, export, secret, or runtime data was inspected.

## User-Performed Streamlit Manual Checks

None recorded. A user must still validate the checklist in `docs/checklists/regression_checklist.md`, including Saved/Unsaved/Save, dirty Reload Keep/Discard, header refresh, paper isolation, non-note actions, browser refresh/application restart, and PDF usability after Reader actions.

## Not Performed

- Browser-level Streamlit Reader smoke.
- Browser refresh and application restart persistence checks.
- Visual PDF usability/rerender observation after Reader actions.

## Accepted Limitations and Deferred Work

- Streamlit may rerun the full script and rerender the PDF after widget interactions. v1.0.24 removes avoidable application-triggered reruns but does not attempt to defeat the framework model.
- G4 Reader stability remains conditional until user manual smoke passes.
- Full PDF rerender resolution is deferred to a future frontend/PDF.js vertical slice.
- Tag governance and quality expansion are frozen during the v1.0.24-v1.0.25 stabilization sequence.
- Lifecycle/recovery closure continues in v1.0.25 on the same branch.
- FastAPI/frontend implementation, autosave, storage/identity changes, and all other stated non-goals remain out of scope.
